# SwipeFlow

SwipeFlow 是一个独立、轻量、面向竖向视频流体验的原生播放器项目。第一阶段以 macOS 为目标，后续可评估 iOS、iPadOS 和 Android。

项目重点不是把网页包装成桌面应用，而是集成原生播放器内核，实现更广的视频编码适用性、可拖动进度控制、播放器复用、提前缓冲，以及类似短视频应用的跟手竖向切换。

## 目标

- 使用 libmpv 等原生播放器内核，不依赖浏览器 `<video>` 解码能力
- 支持硬件解码，并为系统不支持的编码提供软件解码回退
- 维护“上一条、当前条、后五条”的播放器与预缓存窗口
- 触控板、鼠标和键盘均可流畅完成竖向切换
- 当前视频自动播放，离屏视频立即暂停并复用播放器资源
- 不转码、不上传视频、不依赖第三方云服务
- 不提供字幕功能，也不自动加载外挂字幕
- 账号密码进入系统钥匙串，不写入源码或普通配置文件
- 生成供本人使用的本地 ad-hoc 签名 macOS `.app`

## 媒体来源

播放器核心与任何特定服务解耦。媒体来源通过连接器提供统一的目录、列表、播放和可选整理能力。

计划中的连接器：

1. 本地视频目录
2. 本地 `.strm` 目录
3. Vidpick 服务
4. 可选的 OpenList 直连模式

每个连接器声明自己的能力。例如本地目录可以浏览和播放；Vidpick 连接器还可以同步喜欢、待删除状态，并在用户最终确认后请求服务器删除原文件。未标记待删除的视频默认保留，不单独设置“保留”状态。

## 与 Vidpick 的关系

Vidpick 是独立的服务器和 Web 整理工具，负责媒体索引、远程状态及安全删除。

SwipeFlow 不依赖 Vidpick 才能运行。连接 Vidpick 时，它只是使用 Vidpick 公开的客户端接口：

- 服务地址与账号绑定为一个连接配置
- 服务地址保存在本机偏好设置中
- 密码保存在 macOS 钥匙串中
- OpenList Token 和 SmartSTRM 服务器路径不会进入客户端
- 播放签名只在内存中短暂使用
- 客户端不会把 Vidpick 登录凭据转发到媒体存储主机

两个项目使用独立仓库、独立版本和独立发布流程。SwipeFlow 不会把 macOS 客户端代码放回 Vidpick 仓库。

## 初步架构

```text
SwiftUI macOS App
├── Feed UI
│   ├── 竖向分页与跟手动画
│   ├── 键鼠和触控板操作
│   └── 整理操作层
├── Playback Core
│   ├── libmpv 播放器适配
│   ├── 播放器池
│   ├── 预解析与预缓存
│   └── 进度、声音与画面控制
├── Media Source Protocol
│   ├── Local Folder
│   ├── STRM Folder
│   └── Vidpick Connector
└── Secure Storage
    ├── Connection Profiles
    └── macOS Keychain
```

## 初步阶段

1. 建立独立 SwiftUI 工程和通用媒体模型
2. 完成本地视频与 `.strm` 目录播放器原型
3. 集成 libmpv，并用浏览器播放异常的视频验证兼容性
4. 实现播放器池、预缓存和竖向切换
5. 添加 Vidpick 服务地址、账号登录和远程媒体连接器
6. 接入喜欢、整理、待删除与最终确认
7. 构建并验证本机 ad-hoc 签名的 `.app`

## 当前开发状态

- 已建立工程模块边界：Swift Package 提供 `SwipeFlowCore`、
  `SwipeFlowConnectors`，Xcode App target 提供 `SwipeFlowApp`
- 已定义媒体来源、能力声明、分页列表、短期播放解析与可选整理操作协议
- 已实现本地视频目录和本地 `.strm` 目录的浏览与播放地址解析
- 已实现可选 Vidpick 连接器：通过 HTTPS 扫描远程视频、在播放前解析
  302 地址，并在跳转处停止认证信息传播
- 已提供 Vidpick 连接界面；非敏感连接配置保存在本机偏好设置，密码只保存
  在 macOS 钥匙串中；启动时自动恢复连接，一次连接会话只读取一次钥匙串
- 已提供 SwiftUI 目录选择、纵向分页 Feed、键盘/滚动切换和原生视频画面；
  macOS 播放区域按窗口与视频原始比例自适应，不再固定为手机竖屏画幅；
  圆角裁切与边框跟随实际视频宽高比，不包围播放器内部的留黑区域；每条视频
  都吸附到页面顶部的同一稳定锚点，不会因横竖屏比例变化而上下漂移
- 侧栏会持续显示当前来源，并提供逐级进入的层级播放目录选择、实时文件名筛选、
  随机排序、重新随机和播放数量统计；切换目录不需要重新连接来源
- 已提供位于视频画面下方的 Liquid Glass 播放控制栏：显示当前/总时长，
  支持进度拖动、播放/暂停、按总时长前后跳转 5%、静音和空格键快捷控制；
  控制栏只在鼠标进入其底部区域时浮现，不遮挡视频内容；应用启动时默认静音，
  用户打开声音后切换视频会继续保持有声，只有 macOS 默认输出设备改变时才会
  重新静音；底部右侧的独立声音按钮始终可见，会明确显示当前处于“静音”还是
  “有声”状态
- macOS 键盘可用上下方向键切换视频，左方向键按总时长后退 5%；右方向键
  短按前进 5%，按住约 0.35 秒后临时以 2× 播放，松开恢复 1×，两种操作
  互斥且不会触发按键提示音；倍速状态只在视频顶部显示小型提示；空格键控制
  播放/暂停，`L` 切换喜欢状态，`D` 切换待删除状态
- 当前视频信息卡保留文件名与完整媒体路径，只在鼠标进入左上角对应区域时显示
- 控制栏提供播放调试信息：显示容器、编码、分辨率、帧率、码率、硬件解码、
  缓存、丢帧、错误和脱敏后的地址跳转；远程加载失败后会以一字节范围请求检查
  后续跳转、HTTP 状态、内容类型和首响应时间，查询参数、Token 与签名不会原样显示
- 本地文件加载等待上限为 15 秒，远程媒体为 45 秒；失败状态可直接重试播放
- 评选状态会优先保存在 SwipeFlow 本机；评选只包含“喜欢”和“待删除”，
  其余视频默认保留，侧栏可按文件名查看具体记录；侧栏固定提供“随机播放我的
  喜欢”播放列表入口，数量会随评选结果实时更新
- 已实现 Vidpick 风格的喜欢、待删除、统计与逐项撤回；最终
  删除只对声明支持永久删除的来源开放，并要求复核清单后二次确认；播放页的
  “待删除”按钮和 `D` 键均为原地切换，再按一次即可撤回，不会自动跳到下一条
- 已实现“上一条、当前条、后五条”的播放器复用、预加载和界面协调层；
  每个播放器以内存方式预读约 15 秒，并以 64 MiB 限制单路缓存上限
- 已建立标准 macOS App 工程，包含沙盒、只读目录访问、出站网络权限和本机
  ad-hoc 签名配置
- 已建立独立 `NativePlayback` 子包，完成 libmpv API 版本检查、受控初始化、
  串行异步命令、真实加载事件等待以及加载、播放、暂停、跳转和卸载适配

开发环境使用 Xcode 26.6、Swift 6.3，应用最低支持 macOS 14。核心包和测试：

```sh
swift build
swift test
swift run SwipeFlowChecks
```

首次构建原生运行时需要 `meson`、`ninja` 和 `pkg-config`。脚本下载固定的
上游源码、校验 SHA-256、构建、打包并运行依赖与启动审计：

```sh
Scripts/NativeRuntime/build.sh
```

随后构建本机 arm64、Release、ad-hoc 签名的 `.app`：

```sh
Scripts/build-app.sh
```

产物位于 `.build/XcodeDerivedData/Build/Products/Release/SwipeFlow.app`。
`SwipeFlowChecks` 保留为快速冒烟检查，`swift test` 运行完整 XCTest。

使用已打包的 LGPL-only 运行时单独验证原生播放桥接：

```sh
Scripts/test-native.sh
```

Homebrew 的 mpv 二进制及其 GPL 依赖不会进入 Xcode App target、应用包或公开仓库。
根目录的构建和测试仍不依赖 mpv。

## 发布与许可证

本项目计划作为独立 Git 仓库公开发布。源代码不会包含真实服务地址、账号、密码、Token、Cookie、播放签名或服务器路径。

SwipeFlow 自有源代码使用 [MIT License](LICENSE)。libmpv、FFmpeg 及其他第三方组件
继续遵循各自许可证；发布包内会保留对应许可证文件和 LGPL 运行时替换说明。

播放器依赖引入前必须完成 libmpv、FFmpeg 及其构建选项的许可证核对；第三方二进制不会在许可证不明确的情况下直接提交到仓库。
